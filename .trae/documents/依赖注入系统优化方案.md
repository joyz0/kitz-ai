# 依赖注入系统优化方案

## 现状分析

### 当前依赖注入系统
- **中心化容器**：使用 `defaultContainer` 作为全局依赖容器
- **复杂组件**：包含容器、注册器、生命周期管理器、解析器等多个组件
- **依赖解析**：通过字符串或符号键解析依赖
- **生命周期管理**：支持单例、原型、请求、会话等多种生命周期

### 存在的问题
1. **复杂性高**：需要维护多个组件和生命周期管理
2. **依赖关系不明确**：通过容器解析依赖，代码可读性差
3. **测试困难**：需要设置和模拟整个容器
4. **维护成本高**：中心化系统增加了理解和维护的难度

### 配置模块的实现优势
- **简单直接**：通过函数参数传递依赖
- **类型安全**：使用 TypeScript 类型定义依赖
- **易于测试**：可以轻松模拟依赖
- **依赖关系明确**：函数签名直接显示依赖

## 优化方案

### 1. 采用函数参数式依赖注入
- **核心思想**：将依赖作为函数参数传递，而不是通过中心化容器解析
- **实现方式**：为每个模块定义依赖类型，使用默认值处理可选依赖
- **示例**：参考配置模块的 `ConfigIoDeps` 模式

### 2. 减少中心化容器依赖
- **逐步迁移**：优先迁移新代码和核心模块
- **保留容器**：为兼容性保留容器，但减少其使用范围
- **模块解耦**：每个模块独立管理自己的依赖

### 3. 增强类型安全性
- **严格类型定义**：为所有依赖定义明确的类型
- **编译时检查**：利用 TypeScript 的类型系统确保依赖正确性
- **依赖验证**：在运行时验证必需的依赖

### 4. 简化测试流程
- **模拟依赖**：通过参数传递轻松模拟依赖
- **单元测试**：每个模块可以独立测试
- **集成测试**：减少对容器的依赖，测试更直接

### 5. 文档和最佳实践
- **更新文档**：提供新的依赖注入最佳实践
- **代码示例**：展示如何使用函数参数式依赖注入
- **迁移指南**：帮助团队成员理解和迁移到新方式

## 实施步骤

1. **评估现有代码**：识别使用中心化容器的关键模块
2. **优先迁移**：从核心模块开始，逐步迁移到函数参数式依赖注入
3. **更新工具函数**：提供辅助函数简化依赖管理
4. **测试验证**：确保迁移后的代码功能正常
5. **文档更新**：更新相关文档和示例

## 预期效果

- **代码可读性提高**：依赖关系一目了然
- **维护成本降低**：减少对复杂容器系统的依赖
- **测试效率提升**：更容易编写和维护测试
- **开发速度加快**：简化的依赖管理流程
- **系统可靠性增强**：减少依赖解析错误
