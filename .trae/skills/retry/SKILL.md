# 重试与容错 Skill

## Triggers

### When to use
- 当需要处理可能临时性失败的外部服务调用时
- 当需要提高系统稳定性和可用性时
- 当需要防止服务雪崩和级联失败时
- 当需要为异步操作提供超时控制时

### When NOT to use
- 当操作是幂等的，重试可能导致重复执行时
- 当错误是永久性的，重试无意义时
- 当需要实时响应，无法接受重试延迟时
- 当操作涉及敏感操作，不允许重复执行时

## Input

| Parameter | Type | Description | Required |
|-----------|------|-------------|----------|
| attempts | number | 最大尝试次数 | false |
| minDelayMs | number | 最小延迟时间 | false |
| maxDelayMs | number | 最大延迟时间 | false |
| jitter | number | 抖动比例 (0-1) | false |
| shouldRetry | function | 自定义重试判断函数 | false |
| timeoutMs | number | 操作超时时间 | false |
| circuitBreakerOptions | object | 熔断器配置 | false |

## Output

| Parameter | Type | Description |
|-----------|------|-------------|
| result | any | 操作成功的结果 |
| error | Error | 最终失败的错误 |
| retryCount | number | 实际重试次数 |
| circuitBreakerState | string | 熔断器状态 |

4. **实现超时控制**

   - 测试超时控制
   - 测试错误处理

## Failure Strategy

- **重试失败**：达到最大重试次数后，抛出原始错误
- **熔断器打开**：当失败次数达到阈值时，熔断器打开，拒绝后续请求
- **超时失败**：操作超时时，抛出超时错误
- **错误分类**：根据错误类型决定是否重试
- **资源保护**：避免重试风暴和资源耗尽

## 核心理念

**失败是常态，优雅降级**

- 指数退避 + 抖动
- 基于错误类型的智能重试
- 超时控制
- 熔断机制

## 依赖关系

- **依赖模块**：日志与可观测性
- **被依赖模块**：网关通信协议、媒体处理架构、内存管理系统

## 最佳实践

1. **默认重试**：所有外部调用默认启用重试
2. **智能退避**：指数退避 + 抖动避免雪崩
3. **错误分类**：区分可重试和不可重试错误
4. **熔断保护**：防止故障服务被持续调用
5. **超时控制**：所有异步操作设置超时

## 性能考量

- **重试风暴**：大量并发请求同时重试可能导致服务雪崩
  - 解决：使用抖动（jitter）和指数退避
  - 避免：固定延迟重试或无限制重试
- **内存占用**：熔断器状态和重试队列占用内存
  - 建议：设置合理的缓存大小和 TTL
  - 监控：跟踪熔断器状态和重试次数

## 安全考量

- **重试循环**：避免无限重试导致的资源耗尽
- **敏感信息**：确保重试过程中不泄露敏感信息
- **错误处理**：确保重试失败后有合理的错误处理
- **审计日志**：记录重试事件以便排查问题

## 扩展点

- **自定义重试策略**：支持基于业务逻辑的重试策略
- **分布式熔断器**：支持分布式环境下的熔断器状态同步
- **监控集成**：集成 Prometheus 等监控系统
- **自适应重试**：根据系统负载自动调整重试策略
